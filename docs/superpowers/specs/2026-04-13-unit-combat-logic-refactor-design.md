# 单位战斗逻辑重构设计

## 目标
将当前单位的索敌 → 移动 → 攻击整链路重构为一套更可控、更低算力、适合手机端微信小游戏运行环境的轻量行为系统，同时优先消除原地抖动、鬼畜横跳和大量单位挤成一团不出手的问题。

## 范围
本次设计只覆盖单位行为逻辑重构，不包含战斗画面表现层重做、复杂编队系统、复杂寻路、技能系统扩展或高真实性碰撞解算。

## 核心约束

### 1. 可控性优先
单位行为必须可预测、可调参、可解释。目标有效时不应频繁换目标，接敌后不应因为局部扰动导致反复横跳或无法出手。

### 2. 移动端性能优先
运行环境以手机端微信小游戏为基准。方案必须避免高频复杂邻域扫描、过多位置修正、复杂连续避障和大量随机扰动，优先采用低算力、离散化、规则明确的实现方式。

### 3. 优先解决的坏现象
- 原地抖动 / 鬼畜横跳
- 大量单位挤成一团不出手

## 推荐方案
采用**轻量状态机重构**，并引入**攻击位（engagement slots）**机制。

核心思路：
- 索敌、接敌、占位、攻击分层处理。
- 单位不再直接冲向目标中心点，而是优先争取目标周围的可攻击位。
- 去掉当前导致强不确定性的横向扰动与卡住强推逻辑。
- 保持每个单位在每一时刻只有一个主决策，降低抖动和算力浪费。

## 备选方案

### 方案一：在现有逻辑上持续打补丁
- 改动小。
- 可以暂时缓解部分问题。
- 不推荐，因为当前 `_process_entity()` 已经混合了过多行为，继续叠补丁只会增加不透明度和不可控性。

### 方案二：轻量状态机 + 攻击位
- 可控性高。
- 算力成本可控。
- 最适合作为第一阶段重构方案。
- 推荐采用。

### 方案三：完整编队/槽位系统
- 稳定性更强。
- 开发成本更高，容易过度设计。
- 不作为当前阶段方案，可作为未来扩展方向。

## 当前逻辑框架梳理

### 调度层：`battle_controller.gd`
负责：
- 初始化实体与队伍
- 驱动 `tick_combat()`
- 保存最近战斗事件与状态报告
- 同步逻辑状态到视图层

当前职责整体合理，可以继续作为行为总控。

### 行为层：`battle_simulation.gd`
当前在 `_process_entity()` 中混合执行：
- 目标校验与重选
- 接敌前排斥
- 朝目标移动
- 同队推挤
- 卡住强制 nudging
- 进入攻击态并攻击

这是当前最大问题源：行为步骤混杂，难测、难控、难扩展。

### 结算层：`attack_resolver.gd`
当前只负责基础伤害与命中结算，职责相对单一，建议保留。

## 第一阶段推荐状态机

### 状态定义
| 状态 | 含义 |
|---|---|
| `Idle` | 无目标，停止 |
| `Acquire` | 需要选择目标 |
| `Chase` | 已有目标，正在接近目标攻击位 |
| `Engage` | 已进入攻击位，等待攻击时机 |
| `Attack` | 攻击条件满足，执行出手 |
| `Retarget` | 目标失效、位点失效或不可达，准备重选 |

### 状态流转
- `Idle` → `Acquire`：发现可选目标
- `Acquire` → `Chase`：获得目标与攻击位
- `Chase` → `Engage`：到达攻击位
- `Engage` → `Attack`：攻击 CD 就绪且目标有效
- `Attack` → `Engage`：攻击完成，继续维持接敌
- 任意状态 → `Retarget`：目标死亡、超出锁敌规则、位点失效、长时间不可达
- `Retarget` → `Acquire`：重新选目标或重新选位

## 目标选择规则

### 锁敌原则
- 当前目标只要仍有效，就继续保持。
- 不因为“临时出现更近敌人”而立即切换。
- 切换目标只能由硬条件触发。

### 目标失效硬条件
- 目标死亡
- 目标超出明确锁敌距离
- 目标长期不可达
- 当前目标周围所有攻击位长期不可用，且换位也失败

### 不采用的规则
- 每帧重新扫描并切换到“最优目标”
- 基于轻微距离差频繁切目标

## 攻击位设计

### 第一阶段方案
每个目标只提供 4 个基础攻击位：
- 左
- 右
- 前
- 后

### 设计目的
- 避免多个单位挤向同一中心点
- 降低局部推挤与排斥冲突
- 提高“接敌后能稳定出手”的概率
- 用离散位点替代连续避障，降低算力成本

### 位点规则
- 单位申请攻击位
- 占用成功后朝该位点移动
- 位点被占用则尝试下一个可用位
- 长时间卡住先换位点，再考虑换目标
- 目标死亡或失效时释放位点

## 移动规则重构

### 保留的移动目标
单位主运动只朝“攻击位”推进，而不是朝目标中心乱冲。

### 需要移除或弱化的现有逻辑
| 当前逻辑 | 处理建议 | 原因 |
|---|---|---|
| `lane_bias` | 删除 | 容易导致横向漂移和不可预测路径 |
| 卡住后的 `nudge` | 删除 | 容易造成抖动、横跳、抽搐感 |
| 多层 repulsion 与 push 叠加 | 大幅简化 | 容易让多个修正互相打架 |
| 高频局部排斥主导移动 | 不采用 | 与“可控 + 低算力”目标冲突 |

### 推荐保留的轻量修正
- 极轻量防重叠
- 极少量局部让位
- 必须确保这些修正永远从属于主运动，而不能反向主导路线

## 攻击规则重构

### 当前可保留部分
- 攻击 CD 概念
- `AttackResolver` 负责伤害结算
- 攻击事件回传到表现层

### 需要强化的规则
- 到位才能攻击
- 攻击状态必须比当前更稳定
- 进入攻击位后，不再因为无关位移修正反复掉出攻击逻辑

## 代码层面的第一阶段拆分

### 保留的文件结构
| 文件 | 处理 |
|---|---|
| `battle_controller.gd` | 保留为总控 |
| `battle_simulation.gd` | 保留主文件，但内部拆步骤 |
| `attack_resolver.gd` | 保留独立结算职责 |
| `push_resolver.gd` | 第一阶段弱化使用，后续视情况移除 |

### `battle_simulation.gd` 内部推荐拆分步骤
- `_update_target_state()`：校验当前目标是否仍有效
- `_acquire_target()`：选择目标
- `_resolve_engagement_slot()`：分配/确认攻击位
- `_move_to_engagement_slot()`：向攻击位移动
- `_can_attack_target()`：判断是否能出手
- `_perform_attack()`：执行攻击和事件记录
- `_release_engagement_slot_if_needed()`：目标失效时释放位点

第一阶段不强制拆成多个新脚本，避免过早模块爆炸。

## 第一阶段不做的内容
- 复杂编队系统
- 复杂路径搜索
- 高真实性碰撞与群体避障
- 多兵种差异化站位系统
- 技能驱动的高级索敌规则

## 测试策略

### 必测行为
| 类别 | 目标 |
|---|---|
| 索敌稳定性 | 当前目标有效时不频繁切换 |
| 攻击位分配 | 多单位围同一目标时能分配到不同位点 |
| 位点切换 | 位点不可达时先换位，而不是乱跳 |
| 攻击可达性 | 到位后能稳定进入攻击 |
| 抖动回归 | 连续多 tick 后不能横跳/震荡 |
| 拥挤场景 | 不能大面积挤成一团不出手 |

### 不建议的测试方式
- 只看最终是否击杀
- 依赖随机结果判断行为正确性
- 用视觉结果代替逻辑行为断言

## 第一阶段验收标准
- 单位在目标有效时不会频繁切换目标
- 单位接敌时不再明显原地抖动或鬼畜横跳
- 多单位围同一目标时，不再大量挤成一团不出手
- 行为逻辑单 tick 的运算复杂度不明显高于当前实现
- 适合后续继续扩展为兵种差异或更复杂站位系统

## 后续扩展方向
- 从 4 攻击位扩展为 6/8 位攻击位
- 不同兵种使用不同攻击位半径和优先级
- 针对远程/近战拆分不同接敌规则
- 进一步把目标选择和攻击位分配从 `battle_simulation.gd` 中独立出去
